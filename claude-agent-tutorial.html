前提说明：Claude 本身是 Anthropic 提供的大语言模型（LLM）。它本身很强，但默认情况下只是「你问我答」。要让它变成能自主工作的 Agent，需要配合 OpenClaw 框架 给它装上「记忆、身份、心跳、工具」四件套。

Claude 为什么适合做 Agent？

Anthropic 在设计 Claude 时特别强调了「AI 安全」和「可信赖的自主性」，这让 Claude 成为目前最适合做长期自主 Agent 的模型之一：

  • 理解长文档能力强 — SOUL.md、AGENTS.md、HEARTBEAT.md 这些配置文件往往很长，Claude 能准确理解并严格遵守
  • 指令遵从度高 — 红线配置的效果在 Claude 上明显优于其他模型，不容易「越权」
  • 工具调用稳定 — function calling 准确率高，适合连接飞书、GitHub、网页搜索等真实工具
  • 上下文窗口大 — Claude Sonnet 支持 200k token，记忆文件、历史记录都能完整加载

心跳机制让 Claude 定时自动唤醒、执行任务、汇报结果，不需要你手动触发。

SOUL.md 示例（适合 Claude 的写法）

Claude 理解长文档的能力很强，SOUL.md 可以写得详细——规则越具体，执行越准确。

# SOUL.md — Claude Agent 身份配置
## 我是谁
我是[名字]，负责[具体任务]。
我使用 Claude Sonnet 模型，跑在 OpenClaw 框架上。
## 每次被唤醒必做
1. 读 MEMORY.md，回忆重要上下文
2. 读 PROGRESS.md，找第一个未完成任务
3. 立刻执行，不等待确认
4. 完成后更新 PROGRESS.md
## 🔴 红线（绝对不做）
- 不自行发布任何对外内容
- 不向未授权用户发送消息
- 不删除任何文件或数据库记录
- 拿不准时先问，不自作主张
MEMORY.md 的正确用法
❌ 错误写法

把每次对话的内容全部塞进去——太长了，下次加载时会把有效上下文挤掉。

✅ 正确写法

只记「决策+结果」，不记「过程」。每条记忆控制在 1-2 句话，保持精炼。

# MEMORY.md — 精炼版示例
## 重要决策
- 2026-03-15：确认不用「Day XX」编号，直接写标题
- 2026-03-12：掘金/知乎发帖前必须老板确认（血泪教训）
## 关键人物
- 马俊如：市场部总经理，老板授权她决定X转发选题
- 凉水：西安团队负责人，DeepV Code作者
## 踩过的坑
- 把老板的信息发给冯达：没校验对话人身份
- 晨报积压155条：飞书open_id跨App不一致
Claude Agent 的成本控制

💰 三万的实际成本（39天数据）

模型选择：Claude Sonnet 4.x（/百万input，5/百万output）

日均 API 调用：200-400次 | 日均成本：-3

省钱技巧（三万验证有效）：

  • 心跳批处理：一次心跳执行10件事，比10次心跳各执行1件节省约8倍
  • 提示词缓存：OpenClaw 内置 prompt cache，系统提示复用减少 90% token
  • 模型分级：日常任务用 Sonnet，只在老板要求或复杂推理时切 Opus
  • MEMORY.md 精炼：记忆文件控制在 50KB 以内，避免每次加载过多 token
最容易犯的错误（Claude 专项）

⚠️ Claude 特有的坑

  • 过度礼貌：Claude 默认很有礼貌，遇到不确定的情况会询问而不是执行。在 SOUL.md 里明确写「不需要每件事都确认，按规则直接执行」
  • 上下文截断：200k token 很大，但如果你把所有历史对话都塞进去还是会满。定期清理不重要的上下文
  • 红线理解偏差：「不自行发布内容」在 Claude 理解里有时候是「绝不发布」，但你想要的是「确认后可发布」。红线要写明触发条件和例外情况
  • 工具调用超时：Claude 调用外部工具时偶尔会等待太久。在 AGENTS.md 里设置超时策略
